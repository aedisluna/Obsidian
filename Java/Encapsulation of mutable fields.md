## 🔹 Инкапсуляция изменяемых полей

### Проблема

Если поле — изменяемый объект (например, `List`), то даже при `private` можно сломать инкапсуляцию.

```java
private List<Integer> grades;
```

Если вернуть его через getter — внешний код сможет изменить внутреннее состояние.

---

### ❌ Плохой вариант

```java
public List<Integer> getGrades() {
    return grades;
}
```

➡️ Любой код может:

```java
student.getGrades().add(100);
```

➡️ Нарушается инвариант класса

---

### ✅ Инвариант

Инвариант — это правила, которые объект должен всегда соблюдать.

Пример:

- оценки только от 2 до 5
    

---

### ✅ Решения

#### 1. Defensive copy

```java
return new ArrayList<>(grades);
```

✔ внешний код не влияет на внутренний список  
❌ можно менять копию

---

#### 2. Unmodifiable view

```java
return Collections.unmodifiableList(grades);
```

✔ нельзя изменить список  
✔ сохраняется контроль внутри класса

---

### 💡 Когда что использовать

|Подход|Когда использовать|
|---|---|
|Copy|если нужно дать изменяемую копию|
|Unmodifiable|если список только для чтения|

---

### 🔥 Главное

`private` ≠ безопасно  
Важно не отдавать наружу mutable-объекты
